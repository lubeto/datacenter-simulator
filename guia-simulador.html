🖥 Terminal — investigación y detección

netstat -anLista todas las conexiones activas. Busca IPs desconocidas, muchas SYN_RECV = DDoS.
tcpdump -i eth0Captura paquetes en tiempo real. Paquetes SYN masivos desde la misma IP = ataque de flood.
ps auxLista procesos. Si ves flood.sh, hydra o nmap = proceso malicioso activo.
free -mUso de memoria en MB. RAM >85% puede indicar memory leak o proceso malicioso.
df -hUso de disco. >90% puede indicar un ataque de llenado de logs o fallo de disco.
tail -f /var/log/syslogÚltimas líneas del log del sistema. Muestra errores del kernel y eventos de seguridad.
systemctl status nginxEstado de un servicio (nginx, sshd, haproxy…). Muestra si está activo o caído.
systemctl restart nginxReinicia el servicio. Útil para recuperar servicios caídos por sobrecarga.
ping WEB-01Prueba conectividad a un nodo. Timeout = nodo offline. Latencia alta = saturación.
iptables -LLista las reglas del firewall desde la terminal. Confirma si tu bloqueo está aplicado.
iptables -A INPUT -s 203.0.113.45 -j DROPBloquea una IP específica desde la terminal (alternativa al panel Firewall).

Parte 2 — Procedimiento por tipo de ataque

🌊
DDoS / SYN Flood
Red — Ataque externo
1
Terminal: netstat -an — busca cientos de conexiones en estado SYN_RECV desde IPs 203.0.113.X
2
Terminal: tcpdump -i eth0 — confirma paquetes SYN masivos desde la misma IP
3
Logs: access.log — marca como IOC las líneas con esa IP. Usa el botón "→ FW" para copiarla
4
Firewall: bloquea la IP exacta. El nodo cambia a verde en el mapa
5
Verificar: netstat -an de nuevo — el número de conexiones debe haber bajado
Pista clave: más de 500 conexiones activas + estado SYN_RECV = DDoS en progreso
🔑
Brute Force SSH
Red — Ataque externo
1
Logs: abre auth.log — busca líneas con Failed password for root repetidas desde la misma IP 198.51.100.X
2
Terminal: netstat -an — verás muchas conexiones ESTABLISHED en el puerto 22 desde esa IP
3
Terminal: ps aux — puede aparecer el proceso hydra marcado como sospechoso
4
Firewall: bloquea la IP atacante. Opcionalmente bloquea el puerto 22 si el ataque persiste
Pista clave: decenas de "Failed password" en auth.log = alguien está probando contraseñas automáticamente
🔍
Port Scan
Red — Reconocimiento
1
Terminal: tail -f /var/log/syslog — busca nf_conntrack: table full, señal de escaneo masivo
2
Terminal: tcpdump -i eth0 — verás SYN a decenas de puertos distintos desde la misma IP 198.51.100.X
3
Terminal: ps aux — puede aparecer proceso nmap como sospechoso
4
Firewall: bloquea la IP del escáner
Pista clave: conexiones a muchos puertos distintos desde una sola IP = alguien mapea tu red
🧠
Memory Leak
Hardware / Software interno
1
Terminal: free -m — RAM usada >85%, poca memoria libre
2
Terminal: ps aux — proceso leaky_app con CPU y RAM desproporcionados
3
Logs: syslog — línea Killed process (leaky_app): el SO mató el proceso
4
Terminal: systemctl restart del servicio afectado para recuperar el nodo
No hay IP que bloquear — es un problema del software, no un atacante externo
🌡️
Sobrecalentamiento (SST)
Hardware / Físico
1
Panel SST: sensor de temperatura en rojo (>35°C pasillo frío, >45°C rack)
2
Logs: syslog — CPU0: Core temperature above threshold, cpu clock throttled
3
Acción: reportar en el panel guiado. Escalar a mantenimiento físico
No hay IP que bloquear — causa física: fallo de enfriamiento o ventiladores
